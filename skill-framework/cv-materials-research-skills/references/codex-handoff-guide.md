# Codex 交接规范

当某个 Skill 输出“可交给 Codex 的实现任务”时，尽量把任务写成可以直接执行的最小工作单元，而不是抽象口号。

## 每个任务应尽量包含

- 目标：要改什么，或要验证什么
- 作用对象：代码文件、配置、脚本、数据表、图表或论文文件
- 输入：已有实验设定、论文结论、日志、表格、图片或 PDF
- 输出：期望产物，例如新模块、实验配置、统计图、表格、改稿片段
- 验收方式：最快的本地检查、单元测试、脚本命令或对比指标

## 推荐任务写法

### 类型 1：代码实现

- 在指定模型模块里增加某个最小改动
- 保持原训练入口不变
- 补 1 个最小可运行配置
- 先跑最快的 smoke test 或 shape check

### 类型 2：实验配置

- 新建一份独立配置而不是覆盖旧配置
- 明确数据集、输入尺寸、batch size、epochs、seed
- 指定输出目录和结果命名规则

### 类型 3：结果分析

- 从现有日志或 CSV 读取关键指标
- 生成一张指定类型的图或一张对比表
- 标记主结论、异常点和缺失对照

### 类型 4：论文资产

- 生成可直接插入论文的 LaTeX 表格或图注
- 整理 references.bib 或引用占位清单
- 把审稿意见变成逐条改稿 checklist

## 保守原则

- 优先给出“最小可验证改动”，不要一次抛出大重构。
- 对研究代码，先补最便宜的验证，再建议大规模训练。
- 如果任务依赖关键缺失信息，先把缺口写清楚，再输出一个“待补信息后执行”的任务。
- 不要把“想办法提升性能”这种不可验收的描述交给 Codex。
