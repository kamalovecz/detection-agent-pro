# FluidAgent Pro 工作流说明书

## 1. 文档目标

这份说明书不是介绍某一个脚本怎么运行，而是回答两个更核心的问题：

1. 你已经把项目框架搭起来了，后面怎样把它真正变成“从零到一可持续使用”的研究工程。
2. 当你手里只有一个选题方向，或者只有一篇论文初稿时，怎样把这些材料放进当前工程结构里，让 `FluidAgent Pro` 能稳定接手。

这个项目当前更适合以下研究链路：

`选题整理 -> 方案实现 -> 训练/验证 -> 结果分析 -> 论文草稿 -> 模板导出 -> 人工审核回路`

从代码和现有目录来看，控制器真正依赖的核心输入是：

- `fluid_agent_pro.py`
- `research_plan.md`
- `metadata.json`
- `references.bib`
- `paper-template/`

控制器会自动创建或持续写入的阶段性输出主要是：

- `analysis/`
- `logs/`
- `plots/`
- `runs/`
- `paper.typ`
- `paper_final.typ` / `paper_final.tex`
- `paper.pdf`

## 2. 你应该怎样理解这个工程

不要把它理解成“一个单次运行脚本”，而要把它理解成“一个研究工作区控制器”。

它的核心价值不在于替你写一段代码，而在于把研究过程拆成固定阶段，并且在阶段之间插入人工审核点，避免出现下面这些常见问题：

- 代码还没跑通就开始写论文。
- 指标还没核实就开始做图。
- 图表和正文不一致。
- 草稿已经写完，但没有模板化导出路径。
- 一旦推翻某个实验，整个工作区的文件会混乱。

所以你后续维护这个项目时，重点不是“再加几个脚本”，而是把每个阶段的输入、输出、验收标准固定下来。

## 3. 两种进入方式

### 3.1 从“选题方向”进入

这是最标准的用法，适合你现在要从零启动一个新研究主题。

推荐顺序：

1. 先把研究问题写进 `research_plan.md`，不要一上来就写代码。
2. 把论文题目、作者、关键词、基础 DOI 等信息写进 `metadata.json`。
3. 把已有参考文献整理到 `references.bib`，如果 Zotero 已经整理过，再同步补充。
4. 在 `src/` 放入你的基线工程与改进模块。
5. 在 `configs/` 放训练、数据集、验证配置。
6. 在 `weights/` 放预训练权重和阶段性最佳权重。
7. 跑通 Phase 0 到 Phase 2，再进入论文草稿阶段。

这种方式的好处是：

- 研究问题先定，工程组织更稳。
- 实验、分析、写作三条线不会互相打架。
- 后面更容易做消融、复现实验和模板导出。

### 3.2 从“论文初稿”进入

如果你已经有一篇草稿，或者有一篇要大修的 PDF/Word/文本，也可以从初稿反推工程。

推荐顺序：

1. 把原始论文材料放进 `origin_paper/`。
2. 把草稿中的方法、实验表、参考文献缺口拆成可执行任务。
3. 用 `research_plan.md` 重写成工程阶段，不要直接围着初稿逐段修。
4. 把缺失的实验补到 `src/`、`configs/`、`analysis/`、`plots/`。
5. 等实验与图表重新闭环后，再把结果灌回 `paper.typ` 或模板导出流程。

你当前的 `origin_paper/` 已经很适合这类场景，里面可以放：

- 原始 PDF
- 全文抽取文本
- 候选参考文献
- 引文映射
- 修订 checklist
- 表格骨架

这意味着它更像“论文侧证据仓”，而不是“最终输出目录”。

## 4. 推荐的完整流程

下面这套流程是最适合你当前项目结构的。

### Phase 0：定义任务，不要先写代码

这一阶段主要操作：

- 明确研究问题。
- 写 `research_plan.md`。
- 确认环境、依赖、模板、参考文献是否齐全。

你在 `research_plan.md` 里至少要明确：

- 研究目标是什么。
- Phase 1 要产出什么代码和验证结果。
- Phase 2 要产出什么分析表和图。
- Phase 3 要产出什么论文草稿和模板结果。
- 每个阶段怎么判定“可以进入下一步”。

如果这一步写不清楚，后面一定会出现“代码很多，但不知道哪次实验能写进论文”的问题。

### Phase 1：实现与验证

这一阶段的目标不是“多写代码”，而是得到可复查的实验产物。

你应该把下面这些东西组织好：

- `src/`：模型、训练、验证、推理代码。
- `configs/`：数据集配置、模型配置、训练配置。
- `weights/`：预训练权重、最佳权重、导出权重。
- `logs/`：训练日志、验证日志、导出日志。

这一阶段结束时，至少应该有：

- 可以运行的训练/验证脚本
- 明确的实验配置
- 可核对的日志
- 最终权重
- 可供 Phase 2 使用的指标汇总入口

### Phase 2：分析与对比

这一阶段不要再把“分析”混在训练脚本里。

推荐职责分工：

- `analysis/`：统计、汇总、对比、表格生成脚本
- `plots/`：论文图、对比图、消融图、失败案例图
- `logs/`：原始运行记录，不作为最终图表来源

正确做法是：

1. 从 `logs/` 和实验输出中提取结构化结果。
2. 写入 `analysis/*.csv` 或 `analysis/*.json`。
3. 再从 `analysis/` 生成 `plots/`。

这样做的好处是：

- 训练和分析解耦。
- 改图不需要重跑训练。
- 论文表格可以直接追溯到分析结果文件。

### Phase 3：论文写作与模板导出

这一阶段的输入不应该再是“记忆中的结果”，而必须是前两阶段已经落盘的事实。

当前工程里建议这样组织：

- `paper.typ`：当前写作草稿
- `paper_final.typ` / `paper_final.tex`：模板化导出后的最终稿
- `paper.pdf`：最终导出的 PDF
- `paper-template/`：模板目录，目前已有 `clear-iclr`

引用路径上，优先使用：

- 工作区根目录的 `references.bib`

因为从代码看，模板导出时会优先寻找根目录 `references.bib`，找不到才回退模板自带 `main.bib`。

## 5. 当前项目中每个核心文件夹怎么用

下面这部分是你后续真正会反复查的内容。

### `fluid_agent_pro.py`

这是控制器主入口，不是实验代码目录。

它负责：

- 读取 `research_plan.md`
- 初始化工作区
- 驱动状态机
- 记录阶段产物
- 触发验证与模板导出

你一般不应该把具体实验逻辑都塞进这里，而应该让它保持“调度器”角色。

### `research_plan.md`

这是整个工程的路线图，也是最关键的人工输入之一。

建议你把它当成：

- 项目阶段定义文件
- 阶段目标说明书
- 验收标准文档

不是把它写成空泛计划，而是写成“下一阶段应该产生哪些文件”的文档。

### `metadata.json`

这是论文级元数据，不是训练配置。

适合存放：

- 论文标题
- 作者
- 单位
- 关键词
- 参考 DOI

不建议把实验参数塞到这里。

### `references.bib`

这是最终论文导出优先使用的参考文献入口。

建议用途：

- 保存已经清洗好的正式 BibTeX
- 保持 cite key 稳定
- 给 `paper.typ` / `paper_final.tex` 使用

如果你平时用 Zotero 管理文献，推荐做法是：

1. Zotero 做主文献库。
2. 根目录 `references.bib` 做当前项目的“精简参考包”。
3. 每次只把这篇论文真正会引用的条目同步进来。

### `references_groups.md`

这是论文写作辅助清单，不是正式引用源。

适合用来：

- 按主题查看文献分组
- 快速决定 Related Work 结构
- 检查是否缺少数据集/综述/方法/基线条目

### `src/`

这是你的实验实现主目录。

建议只放：

- 模型代码
- 训练脚本
- 验证脚本
- 推理脚本
- 数据处理逻辑

不建议把论文草稿、日志分析、手工笔记混进来。

### `configs/`

这是实验配置目录。

建议存放：

- 数据集路径配置
- 模型结构配置
- 训练超参数配置
- 导出和部署配置

尽量做到“换配置，不改代码”。

### `weights/`

这是模型权重目录。

建议分清：

- 预训练权重
- 微调权重
- 最优权重
- 导出权重

同时建议在日志或 `analysis/experiment_manifest.json` 中记录权重来源，不要只靠文件名记忆。

### `analysis/`

这是结构化分析目录。

适合存放：

- 主结果汇总
- 消融结果
- 迁移实验结果
- 部署评估结果
- 汇总脚本

它的核心定位是“论文证据中间层”。

### `plots/`

这是论文图表目录。

适合存放：

- 主结果图
- 消融图
- 迁移对比图
- 精度-速度图
- 失败案例图

建议规则：

- 所有图都应由 `analysis/` 生成
- 不要手工截图后直接丢进这里作为最终版本

### `logs/`

这是原始运行记录目录。

适合存放：

- 环境检查日志
- 训练日志
- 验证日志
- 导出日志
- 错误日志

它是“原始证据层”，不是最终论文图表层。

### `runs/`

这是按状态机阶段保存的过程目录。

从代码看，每次阶段运行都会在这里按状态和时间戳落盘。

适合用来：

- 回看某一轮尝试的 prompt、输出、报告
- 对比不同阶段的生成结果
- 追踪失败原因

如果你后面想复盘“为什么这次导出失败”或者“这一轮分析是从哪个输入生成的”，最先查的应该是这里。

### `paper-template/`

这是模板目录，不是论文正文目录。

当前项目里已经有：

- `paper-template/clear-iclr`

控制器会在模板导出阶段自动寻找受支持的模板目录，所以你后续新增模板时，应该沿着现有结构增加，而不是把模板文件散放在根目录。

### `origin_paper/`

这是原始论文材料目录，特别适合“从论文初稿进入”的场景。

你当前已经放了很多很合适的材料，例如：

- 原始 PDF
- 全文文本
- 引文需求清单
- 候选参考文献
- 句子到 bibkey 映射
- 修订 checklist

建议继续保持这个定位：

- 存“原始材料和修订依据”
- 不存“最终导出结果”

### `source-prompts/`

这是提示词参考资产目录。

它更适合：

- 存放写作提示模板
- 存放研究写作参考材料
- 给后续扩展 prompt 流程做素材仓

不建议把正式实验结果放这里。

### `scripts/`

这是辅助脚本目录。

你当前这里已经包含：

- Zotero 相关启动脚本
- Word MCP 启动脚本
- BibTeX 导出脚本

建议今后继续只放“辅助工具脚本”，不要把主训练流程塞进这里。

### `mcp-servers/` 与 `mcp-client-configs/`

这两部分属于外部协作能力，不是论文主流程的实验目录。

你可以把它们理解成：

- `mcp-servers/`：能力后端
- `mcp-client-configs/`：客户端接入示例

例如：

- Zotero 文献检索
- Word 协作
- 其他 MCP 工具链

如果你的主要目标是做研究闭环，这两部分是增强项，不是 Phase 1/2/3 的核心输入。

## 6. 推荐的日常工作方式

最推荐的工作方式不是“想起什么改什么”，而是固定成下面这套节奏。

### 每启动一个新课题时

1. 复制或重写 `research_plan.md`
2. 更新 `metadata.json`
3. 清理并重建 `references.bib`
4. 把基线代码和新改动放进 `src/`
5. 把实验配置放进 `configs/`

### 每完成一轮实验时

1. 确认 `weights/` 是否保存了可复用结果
2. 确认 `logs/` 是否留下原始记录
3. 把结构化结果整理到 `analysis/`
4. 从 `analysis/` 生成 `plots/`
5. 再决定是否更新论文正文

### 每准备进入论文阶段时

1. 检查 `references.bib` 是否已经清洗
2. 检查 `analysis/` 与 `plots/` 是否一致
3. 检查 `paper-template/` 是否是目标模板
4. 再启动草稿生成与模板导出

## 7. 建议的最小命令路径

### 启动工作流

如果你的工作区就是当前根目录，推荐命令：

```powershell
fluid-agent-pro --workspace .
```

如果你想显式指定目录：

```powershell
fluid-agent-pro --workspace D:\Code\paper_pass_skill\fluid-agent-pro
```

### 清理生成产物，但保留源码

```powershell
fluid-agent-pro --clear --workspace .
```

适合以下场景：

- 想重跑流程
- 不想删 `src/`
- 只清阶段输出和缓存

### 连 `src/` 一起清理

```powershell
fluid-agent-pro --clear-hard --workspace .
```

这个命令风险更高，只适合你确认源码也要重置的场景。

## 8. 你后面最应该补强的地方

如果你的目标是把这个项目真正打磨成“从零到一”的稳定工程，最值得继续完善的是下面几块。

### 8.1 固定研究入口模板

建议后续新增一个“课题初始化模板”，至少自动生成：

- `research_plan.md`
- `metadata.json`
- `references.bib`
- `configs/`
- `src/`
- `analysis/`

这样你每开一个新方向，不用手动拼目录。

### 8.2 固定分析结果接口

建议统一要求 Phase 1 的实验都产出同一类结构化文件，例如：

- `analysis/main_results.csv`
- `analysis/ablation_results.csv`
- `analysis/transfer_results.csv`
- `analysis/deployment_results.csv`

这样 Phase 3 才能稳定依赖这些结果，不会每个项目都换字段名。

### 8.3 固定初稿接入规范

建议把“从初稿进入”的路径制度化，例如要求：

- PDF 放 `origin_paper/`
- 全文文本放 `origin_paper/*_fulltext.txt`
- 修订清单放 `origin_paper/revision_checklist.md`
- 引文映射放 `origin_paper/sentence_to_bibkey_mapping.md`

这样以后任何一篇旧稿都能平滑接进这个工程。

### 8.4 固定参考文献同步机制

你现在已经有：

- `references.bib`
- `references_groups.md`
- `scripts/export_zotero_reference_pack.py`

建议后面把它正式纳入写作前检查项：

1. Zotero 管总库
2. `references.bib` 管当前项目精简包
3. `references_groups.md` 管写作分组

## 9. 最后给你的落地建议

如果你现在就要把这个项目用于下一篇论文，我建议按下面的最小动作开始：

1. 先选定“本篇论文的唯一主线”，写进 `research_plan.md`。
2. 确认 `src/` 和 `configs/` 只服务这条主线，不要把多个方向混在一起。
3. 把当前真正会引用的文献压缩到 `references.bib`。
4. 把原始草稿或外部证据统一放进 `origin_paper/`。
5. 所有结果先经过 `analysis/`，再进入 `plots/` 和 `paper.typ`。

如果你做到这五点，这个项目就已经不只是“框架搭好了”，而是已经具备了稳定跑完整研究闭环的基础。
